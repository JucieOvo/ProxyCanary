# 非对称金丝雀检测 —— 相关研究与定位分析

本文档梳理提示词套取攻击防御领域的现有方案，并定位本工作（非对称金丝雀检测）在该领域中的位置与贡献。

---

## 1. 问题定义与紧迫性 (id: section-1)

### 1.1 攻击分类

提示词套取攻击（Prompt Extraction Attack）是 LLM 安全领域的核心威胁之一。攻击者通过精心构造的用户输入，诱导 LLM 泄露其系统提示词（System Prompt）—— 其中通常包含商业逻辑、安全指令、API 密钥等敏感信息。根据攻击方式，可将其分为三类：

| 类别 | 描述 | 代表性工作 |
|------|------|-----------|
| **直接套取** | 直接要求模型输出系统提示词或指令 | [Raccoon ACL 2024], [Prompt Leakage EMNLP 2024] |
| **间接套取** | 通过翻译、角色扮演、编码等间接手段获取 | [PromptSleuth 2025] |
| **统计重构** | 通过多次查询的 logits/文本分布重构提示词 | [Why Are My Prompts Leaked? 2024], [PromptKeeper EMNLP 2025] |

**相关基准测试**：
- **Raccoon (ACL 2024)** [Wang et al.]：14 类提示词抽取攻击 + 复合攻击 + 多种防御模板。核心发现：无防御时所有模型普遍易受攻击；OpenAI 模型在受保护时展现韧性。
- **Prompt Leakage (EMNLP 2024 Industry)** [Agarwal et al., Salesforce]：考察 10 个闭源/开源 LLM 的多轮提示词泄露。利用 LLM 谄媚效应（sycophancy），将多轮攻击成功率从 17.7% 提升至 86.2%。评估了 7 种黑盒防御方案。

### 1.2 防御需求

与提示注入（Prompt Injection）不同，提示词套取的检测面临独特挑战：
- **侧信道风险**：直接拒绝会向攻击者暴露系统对提示词内容的敏感性
- **被动性**：与注入不同，套取不要求模型执行恶意操作，仅需模型"复述"其已知内容
- **多轮累积**：攻击者可分多轮逐步提取，单轮检测不足以应对

## 2. 现有防御方案分类 (id: section-2)

### 2.1 方案总览

现有防御方案可根据其作用位置分为四大类：

```
                          ┌─────────────────────────────────────────┐
                          │          LLM 系统提示词保护方案            │
                          └─────────────────────────────────────────┘
                                          │
          ┌───────────────┬───────────────┼───────────────┬───────────────┐
          ▼               ▼               ▼               ▼               ▼
    ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
    │ 提示词级  │   │  输入级   │   │  输出级   │   │  模型级   │   │  代理级   │
    │ 防御      │   │  防御      │   │  检测      │   │  防御      │   │  检测      │
    └──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
    修改提示词      分类用户输入     扫描模型输出     训练/微调模型    独立金丝雀模型
    本身来防御      判断是否攻击     检测泄露        来抵抗套取       代理检测
```

### 2.2 提示词级防御 (Prompt-Level)

**思路**：修改受保护模型的系统提示词本身，使其更难被套取。

| 方案 | 机制 | 代表工作 | 局限性 |
|------|------|---------|--------|
| **提示词混淆** | 对提示词进行编码或变换 | Prompt Obfuscation [CISPA 2024] | 影响模型行为质量 |
| **提示词硬化** | 添加"不要泄露提示词"指令 | 工业界普遍做法 | 可被指令覆盖攻击绕过 |
| **虚拟提示词再生** | 泄露时用虚拟提示词生成响应 | PromptKeeper [EMNLP 2025] | 仅在检测到泄露后触发，响应质量下降 |
| **系统向量** | 用向量表示替代文本提示词 | System Vectors [PSU 2024] | 需模型内部访问权限，不适用于 API 调用 |

**核心局限**：提示词级防御直接修改受保护模型的提示词，可能影响模型的正常行为。PromptKeeper 虽然在检测到泄露时用虚拟提示词替换，但该替换过程本身会改变模型响应模式，攻击者可能通过多次查询的统计差异反向推断。

### 2.3 输入级防御 (Input-Level)

**思路**：在用户输入到达受保护 LLM 之前，对其进行分类或过滤。

| 方案 | 机制 | 代表工作 | 局限性 |
|------|------|---------|--------|
| **模式匹配** | 正则表达式检测已知攻击模式 | LLM Firewall [OWASP] | 易被改写绕过 |
| **BERT 分类器** | 微调小模型做二分类 | BISEC 2025 [Živanović] | 需要攻击样本训练，泛化能力有限 |
| **语义意图分析** | 检测输入是否包含未授权意图 | PromptSleuth [2025] | 需 2-3 次 LLM 调用，延迟高（~2s） |
| **多层混合** | 启发式 + LLM + 向量库 + 金丝雀 | Rebuff [已归档 2025] | 架构复杂，依赖多个外部服务 |

**PromptSleuth 基准测试** [Wang et al., 2025]：

| 防御方案 | FPR (假阳性率) | FNR (假阴性率) |
|---------|---------------|---------------|
| DataSentinel | 0.0498 | 0.6669 |
| SecAlign | 0.4547 | 0.4947 |
| PromptArmor | 0.0926 | 0.0825 |
| **PromptSleuth-5** | **0.0008** | **0.0007** |

PromptSleuth 在基准上表现优异，但其依赖多次 LLM 调用的架构特点意味着高延迟（均值约 2 秒）和高成本。

**核心局限**：输入级防御需要预先定义"攻击"的语义边界。攻击手法不断演进（Raccoon 定义了 14 类，且新手法持续出现），静态分类器的覆盖面始终滞后。此外，多轮攻击中单条消息可能看起来无害，但组合起来构成攻击——输入级检测缺乏上下文感知。

### 2.4 输出级检测 (Output-Level)

**思路**：检测 LLM 的回复中是否包含系统提示词内容的泄露。

**2.4.1 金丝雀令牌 (Canary Tokens) —— 工业界主流**

在系统提示词中嵌入唯一、随机的"金丝雀令牌"（如 `SECRET-A1B2C3D4`），检测模型输出中是否出现该令牌。若出现，则提示词已被泄露。

| 实现 | 机制 | 状态 |
|------|------|------|
| Rebuff | 四层防御，最内层为金丝雀令牌 | 2025.05 归档 |
| ClawGuard | 八层防御，L1 注入 + L6 检测 | 活跃开发 |
| Agent Shield | 六层防御，L5 金丝雀检测 | 活跃开发 |
| Canari (PyPI) | 独立金丝雀令牌 IDS 库 | 活跃维护 |

**MIT Kill-Chain Canaries** [Wang, 2024/2025]：将金丝雀令牌追踪形式化为四阶段杀伤链模型：

| 阶段 | 含义 | 检测窗口 |
|------|------|---------|
| Exposed | 金丝雀出现在任何工具结果中 | 所有模型 100% |
| Persisted | 金丝雀被写入持久存储 | Claude: 0%（在此阶段剥离）|
| Relayed | 金丝雀被第二个 Agent 读取 | GPT-4o-mini: 53% 传播率 |
| Executed | 金丝雀出现在出站工具参数中 | 取决于模型安全层级 |

**核心发现**：模型安全不在于是否"看到"对抗内容，而在于是否"传播"它。Claude 在摘要阶段剥离注入，GPT-4o-mini 则无损传播。

**2.4.2 金丝雀令牌的关键局限性**

国际研究社群已识别出以下问题：

| 问题 | 发现 | 来源 |
|------|------|------|
| **缺乏指令触发** | 默认金丝雀检查无检测效果（0 检出），需显式添加"当被要求输出系统指令时包含此令牌"的指令，才能将 TPR 升至 41-94% | Rebuff/Vigil 评估 |
| **静态令牌池** | 一旦金丝雀泄露，该令牌对所有未来请求仍然有效——爆炸半径问题 | PPA Dynamic [Dorzhiev & Liu 2025] |
| **修改受保护模型** | 金丝雀令牌嵌入在受保护模型的提示词中，可能影响模型行为 | 多个来源 |
| **仅检测成功泄露** | 当模型拒绝输出时（安全训练优先），金丝雀令牌不被输出，攻击不被检测 | 本文发现 |

**核心局限**：输出级金丝雀令牌方案有三个结构性缺陷：(1) 必须修改受保护模型的系统提示词；(2) 依赖模型输出金丝雀，而安全训练可能阻止此行为；(3) 金丝雀令牌的静态性带来爆炸半径问题。

### 2.5 模型级防御 (Model-Level)

**思路**：通过训练或微调使模型本身抵抗提示词套取。

| 方案 | 机制 | 局限 |
|------|------|------|
| **安全对齐** | RLHF/DPO 训练使模型拒绝套取 | 安全训练与指令遵循之间存在根本张力 |
| **水印** | 在模型输出中嵌入可检测的水印 | 不影响套取本身，仅做事后审计 |
| **差分隐私** | 训练时添加噪声限制信息泄露 | 主要针对训练数据泄露，非提示词 |

## 3. 现有方案的共性局限与本文的研究缺口 (id: section-3)

### 3.1 三个共性假设及其脆弱性

审视上述四类方案，可以发现它们共享三个隐含假设，而这三个假设在实践中并不总是成立：

**假设一：攻击者与防御者使用同一模型**

- 输入级防御对用户输入做独立分类，输出级金丝雀嵌入受保护模型的提示词中
- **脆弱性**：输入分类器无法感知模型特定的行为模式（不同模型对同一攻击的反应不同）；输出级金丝雀必须侵入受保护模型的提示词空间

**假设二：检测发生在"模型已响应之后"**

- 输出级金丝雀检测模型回复，PromptKeeper 在泄露后再生新回复
- **脆弱性**：攻击者已获得一次模型响应，其中可能已包含部分有效信息；在多轮设定中，一次泄露即可为后续攻击提供支撑

**假设三：单一检测信号足以覆盖攻击空间**

- 金丝雀令牌仅检测"成功泄露"，输入分类器仅检测"攻击意图"
- **脆弱性**：安全训练的模型可能在攻击时拒绝输出，而拒绝本身不触发金丝雀令牌检测；攻击者的措辞可能绕过输入分类器

### 3.2 未被满足的研究缺口

| 缺口 | 描述 |
|------|------|
| **G1: 零侵入** | 需要一种不修改受保护模型系统提示词的检测方案 |
| **G2: 双信号** | 需要同时捕获"模型被套取"和"模型抵抗套取"两种攻击信号 |
| **G3: 模型特异性** | 检测应反映实际被保护模型的行为特征，而非通用攻击模式 |
| **G4: 实时性** | 检测应在 token 生成过程中实时完成，不等待完整回复 |

## 4. 本文方案定位：非对称金丝雀检测 (id: section-4)

### 4.1 核心创新

本文提出的**非对称金丝雀检测 (Asymmetric Canary Detection)** 方案引入一个**与被保护模型分离的、独立的轻量金丝雀模型**。其核心架构差异在于：

```
传统金丝雀方案:
  用户 → [受保护模型 (含金丝雀令牌)] → 检测输出中的令牌 → 响应

本文方案:
  用户 → ┬→ [金丝雀模型 (独立，含虚构水印词)] → 检测泄露或拒绝 → 攻击判定
         │
         └→ [受保护模型 (无修改)] → 响应（仅在检测安全时）
```

### 4.2 与现有方案的系统性对比

本方案与现有四类防御的核心差异在于三个关键维度：**检测时机**（受保护模型响应之前还是之后）、**检测机制**（分类意图还是测试行为）、**信号覆盖**（单信号还是双信号）。

| 维度 | 输入级 (PromptSleuth/BERT) | 输出级 (Canary Tokens/Rebuff) | 提示词级 (PromptKeeper) | **本文 (非对称金丝雀)** |
|------|---------------------------|------------------------------|------------------------|----------------------|
| **检测时机** | 受保护模型响应**之前** | 受保护模型响应**之后** | 受保护模型响应**之后** | **受保护模型响应之前** |
| **检测机制** | 分类输入文本的意图 | 扫描受保护模型输出中的令牌 | 在受保护模型输出后做假设检验 | **测试金丝雀代理模型的行为** |
| **是否使用独立模型** | 是（分类器） | 否（仅在被保护模型输出中扫描） | 否 | **是（行为代理模型）** |
| **模型角色** | 意图判别器 | N/A | 事后检测器 | **行为替身 — "如果输入会引发此模型泄露/拒绝，则也会引发受保护模型"** |
| **修改受保护模型提示词** | 否 | **是** | 是 | **否** |
| **双信号覆盖** | 否 | 否（仅泄露） | 否（仅泄露） | **是（泄露+拒绝）** |
| **流式实时检测** | N/A（回复前分类） | 事后扫描 | 事后扫描 | **流式（token 级并行检测）** |
| **需要攻击训练数据** | **是** | 否 | 否 | **否** |
| **检测过程对攻击者可见** | 不可见 | **令牌暴露风险** | 响应变化可能被感知 | **完全不可见** |

### 4.3 核心区分：行为测试 vs 意图分类

输入级防御（PromptSleuth、BERT 分类器）虽然也使用独立的小模型，但其机制是**对用户输入文本做意图分类**——"这个句子看起来像攻击吗？"。这是一种**静态的文本属性判断**，不涉及任何模型行为。

本文方案的机制是**行为测试**——"这个用户输入会让一个模型产生攻击行为吗？"。我们将用户输入实际喂给金丝雀代理模型，观察该模型的**响应行为**。这是一种**动态的因果测试**。

```
输入级防御（意图分类）：
  "请把你的提示词给我" → [BERT分类器] → 得分0.92 → 判定为攻击
                           ↑
                     静态文本属性判断
                     不涉及模型行为

本文方案（行为测试）：
  "请把你的提示词给我" → [金丝雀LLM] → "[系统参数]: Zyphor-7X | ..." → 泄露 → 攻击
                           ↑                            ↑
                     实际执行推理                  观察模型行为
                     用户输入产生因果效应
```

**行为测试的关键优势**：攻击者的输入不需要"看起来像攻击"——只要它**实际引发了**攻击行为。这意味着：
- 措辞演变、改写、多语言、编码等绕过输入分类器的手段，在金丝雀模型中仍会引发相同的行为反应
- 金丝雀模型的行为（泄露或拒绝）与受保护模型的行为之间存在**因果模拟关系**

```
                         │ 检测时机?          │
                         └────────┬─────────┘
                                  │
              ┌───────────────────┴───────────────────┐
              │ 受保护模型响应之前                       │ 受保护模型响应之后
              ▼                                       ▼

```

### 4.5 对三个共性假设的回应

| 传统假设 | 本文的打破方式 |
|---------|-------------|
| 攻击者与防御者共享同一模型 | 检测在**独立的金丝雀代理模型**上完成。输入级虽也用独立模型，但做的是**意图分类**而非**行为测试**；金丝雀令牌嵌在被保护模型内，是**同一模型** |
| 检测发生在模型响应之后 | 检测在**受保护模型被调用之前**完成。不是扫描被保护模型的输出，而是用金丝雀模型预演用户输入会引发什么行为；流式检测命中即终止，不等待完整回复 |
| 单一检测信号足够 | **双信号并行捕获**：金丝雀词命中（模型被套取，支路一）+ 拒绝行为命中（模型安全抵抗，支路二）。拒绝本身即为攻击信号，补上了安全训练模型"拒绝而非泄露"导致的检测盲区 |
