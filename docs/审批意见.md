## 批判性审稿意见

### Guard4PromptAttack: The Canary in the Coal Mine — Proxy Output Inspection for Prompt Extraction Defense

---

### 总体评价

本文提出了一个有创意的视角转换：将提示词套取防御从"分析攻击话术"转向"观察模型行为反应"。这一概念值得探索。论文的自我批判意识较强（主动承认行为传递假设未验证、明确将工作定位为"可行性验证"），在学术诚信方面值得肯定。然而，经过对代码实现、实验数据和相关文献的交叉验证，本文存在多个方法论和实验设计层面的严重问题，**在当前形态下不足以支撑其核心贡献声明**。

建议：**Major Revision（重大修改）**

---

### 一、核心方法论问题

#### 1.1 行为传递假设的致命缺失 [Critical]

论文反复强调"金丝雀模型验证攻击对受保护模型的有效性"，但整个实验设计中**金丝雀模型既是金丝雀也是受保护模型**——实验仅验证了"金丝雀能否检测针对自身的攻击"，完全未涉及"金丝雀检测结果能否预测另一个模型的行为"。虽然论文在 Section 5.3 和 Abstract 中坦率承认了这一点，但这一未验证的前提恰恰是整个方案的**存在理由**——如果金丝雀的行为无法预测受保护模型的行为，那么金丝雀架构相对于直接在受保护模型提示词中加入标记词并无本质优势。

更关键的是，对抗样本领域的跨模型可迁移性（transferability）文献已有大量实证。论文完全未引用此方向的工作来框架化其核心假设（如 Papernot et al., 2016; Tramèr et al., 2017 等）。如果 prompt extraction 攻击的可迁移性与视觉对抗样本类似，那么金丝雀行为与受保护模型行为之间的关联是可以被量化的——论文遗漏了借鉴已有可迁移性理论来论证（或反驳）其核心假设的机会。

**建议**：这不应仅作为"局限性"讨论，而应是一个实验章节。即便使用两个不同规模的 Qwen 模型（如 0.6B 作金丝雀，9B 作受保护模型），也能提供初步的跨模型行为传递证据。作者实际上已有这两个模型的数据，但未设计交叉验证实验。

#### 1.2 "攻击有效性验证"的概念混淆 [Major]

论文将其贡献定位为回答"攻击能不能得逞"（层次二），区别于现有方案回答的"这段输入是不是攻击"（层次一）。但实际实现中，金丝雀模型观察的是**攻击对金丝雀自身**的有效性，而非对受保护模型的有效性。由于行为传递假设未验证，系统实际回答的问题变成了"这段输入能不能让一个拥有特定提示词的模型就范"——这本质上仍然是一种**基于代理的攻击识别**，而非真正的"攻击有效性验证"。

论文的措辞在层次一/层次二的哲学叙述和实际实验能力之间存在显著落差。

#### 1.3 论文声称的零修改优势缺乏实际场景验证 [Major]

论文声称核心优势是"受保护模型零修改"，适用于"第三方 LLM API 或已部署且提示词受合规约束的企业 Agent"。但：
- 所有实验仅在本地 Ollama 上运行 Qwen 模型
- 未在任何第三方 API（GPT-4、Claude 等）上验证
- Section 5.2 自己也承认"上述对第三方 API 场景的适用性论述基于架构推演"

一个以"零修改"为核心卖点的方案，其全部实验都在自己完全控制的环境中进行，这使得核心卖点变成了一个未验证的声明。

### 二、实验设计问题

#### 2.1 数据集构建的方法论缺陷 [Major]

- **攻击样本**：54 条——对于一个声称"系统评估"的工作，这一规模严重不足。3 折交叉验证中每折仅 14 条攻击测试样本，统计效力极低。论文自己也承认"McNemar 检验效力不足"。
- **正常样本**：216 条**由作者自建**，不是来自 Raccoon 基准。论文多处使用"在 Raccoon 基准上完成系统评估"的措辞，但实际只使用了 Raccoon 的攻击部分——正常对话是自己构造的。这一事实在论文中说明不够清晰，容易误导读者认为所有数据均来自 Raccoon。
- **边界案例设计**：53 条边界案例（如"你是怎么工作的"）的选取标准未明确说明。这些边界案例直接影响 TNR 的可信度——如果边界案例设计得不够"硬"，TNR=100% 的价值就会降低。

#### 2.2 基线选取偏弱、不公平 [Major]

代码审查确认所有基线（RegexGuard、KeywordWeight、TF-IDF、LLM-Judge）均为**作者自定义实现**，而非使用现有工具或标准基线库。

- **RegexGuard**（10 条正则）：明显是稻草人基线——没有任何严肃的防御系统仅用 10 条正则。
- **KeywordWeight**：F1=0.441，明显弱于其他所有方法，设置它的目的不清楚。
- **TF-IDF**：实现合理，但作为一个统计方法，它的设计目标本身就不是做攻击检测——拿它当基线有"田忌赛马"之嫌。
- **LLM-Judge**：使用同模型零样本分类，这是一个公平的对比，但缺少 few-shot 或 instruction-tuned 的变体。
- **CanariToken**：**被从 Table 1 中移除**——代码审查发现，移除理由（"3 折中训练/测试集需使用不同的提示词配置"）在技术上站不住脚。完全可以在所有折中使用相同的随机令牌。这个基线恰恰是最相关的竞争者（它直接做输出级令牌验证），被移除后论文的方法就缺少了与最直接竞争者的对比。

**严重缺失的基线**：
- **Llama Prompt Guard**：论文在 Related Work 中提及但未作为基线对比——它是目前最广泛使用的开源 prompt injection 检测器。
- **PromptSleuth**：论文引用其 FNR=0.0007，但未在自己的数据集上复现对比。
- **NeMo Guardrails**：同样仅在 Related Work 中提及。
- **Sentence-BERT / 语义相似度基线**：作为中间复杂度的语义方法，比 TF-IDF 更合理。

#### 2.3 CanariToken 基线被不当移除 [Major]

经代码审查确认，[canari_token.py](file:///F:/Guard4PromptAttack/benchmark/baselines/canari_token.py) 实际已经实现——它生成随机 UUID 令牌嵌入提示词并扫描输出。论文声称因"3 折框架限制"移除它，但技术上完全可以在所有折中固定同一个令牌。这个基线直接对应论文 Section 2 中讨论的"输出级令牌验证"方法，是最直接的竞争者。它的缺席使得读者无法判断 Guard4PromptAttack 的金丝雀架构相对于"直接在受保护模型中嵌入令牌"是否有真正的性能优势。

#### 2.4 TNR=100% 的置信区间问题 [Minor]

论文报告 TNR=100%，Clopper-Pearson 95% CI [90.7%, 100.0%]。38 条测试样本（每折）的置信区间下界仅为 90.7%——这意味着真实 TNR 有 2.5% 的概率低于 90.7%。在生产环境中，TNR<91% 意味着约 9% 的误拦截率，对用户体验的影响是不可接受的。论文对此缺乏讨论。

#### 2.5 多轮上下文免疫实验的设计偏差 [Major]

多轮实验（Section 4.5）的设计存在重要问题：
- 设置 B（"完整上下文"）将多轮对话**拼接为单条消息**发送——这与真实的多轮攻击场景完全不同。真实攻击者是**逐轮发送**消息，模型在每轮都有独立的推理机会。
- 论文也承认了这一点（"设置 B 不等同于真实多轮攻击"），但推理链"如果单条消息内的上下文稀释已经起效，那么多轮积累只会更有效"缺乏理论和实验支持——这两种场景的模型行为机制可能完全不同。
- **仅 10 个场景**作为实验规模严重不足，McNemar 检验的 p=0.046 刚好擦过 0.05 的显著性边界，且样本量如此小时该检验的效力本身就成问题。

### 三、论文撰写问题

#### 3.1 Raccoon 基准使用的描述不够清晰 [Minor]

论文多处使用"在 Raccoon 基准上"的措辞，但实际上只使用了 Raccoon 的攻击部分，正常对话完全由作者自建。应明确说明数据构成，避免 "在 Raccoon 基准上完成系统评估" 这样容易引起误解的表述。此外，Raccoon 原始基准的攻击分类与本文使用的"14 类"划分是否一致，需要明确说明是否进行了重新分类。

#### 3.2 比喻过度使用 [Minor]

"煤矿金丝雀"比喻贯穿全文且多次展开（Introduction、Section 5.1 均有详细类比），占据了较多篇幅。虽然比喻本身形象，但 Section 5.1 中"矿洞塌方""养鸟的成本""鸟笼"等延伸类比更适合科普文章而非学术论文，且在此重复比喻占用了本可用于更多实验分析的空间。

#### 3.3 Related Work 的覆盖遗漏 [Minor-Major]

如前文调查所述，论文遗漏了以下重要相关工作：
- **Tensor Trust** [Toyer et al., NeurIPS 2023]——大规模 prompt injection 攻防数据集
- **Jatmo** [Piet et al., 2024]——任务特定微调防御
- **Signed Prompt** [Suo et al., 2024]——密码学级别的提示词保护
- **对抗样本可迁移性文献**——直接关联核心假设
- 商业解决方案（Lakera Guard、Guardrails AI 等）

其中，对抗样本可迁移性文献的缺失尤为严重——这类工作为"代理模型的行为能否预测目标模型的行为"这一核心问题提供了丰富的理论基础和实证证据。

#### 3.4 引用格式和准确性 [Minor]

- 多数引用仅以 `[姓名, venue 年份]` 形式给出，缺乏完整的书目信息（如卷号、页码、DOI）
- PromptSleuth 的 FNR=0.0007 声明未能通过独立验证
- 部分 2025 年 venue 的论文（如 EMNLP 2025、CCS 2025）的发表状态需确认

### 四、技术细节问题



#### 4.3 非单调效应的解释不足 [Minor]

Table 3 展示了"同一提示词在 9B 和 0.6B 上表现完全相反"的有趣现象，但解释仅停留在"弱模型需要更强指令约束，强模型对抗式语言触发反注入防御"的定性层面。这一发现本应引出更深入的讨论：
- 如果金丝雀模型的选择和提示词需要联合调优，那么方案的实际部署成本远高于论文所暗示的"部署一个金丝雀模型"
- "不存在跨模型的通用提示词"这一发现直接削弱了方案的通用性声明
